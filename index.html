Express ESM • MySQL • React • Axios • Zod • React Hook Form

Architecture Générale

1. Vision globale

Le frontend envoie une requête HTTP vers le backend. Le backend valide, traite et communique avec la base de données.

2. Flux complet

  1. Utilisateur remplit formulaire
  2. Validation frontend (Zod)
  3. Envoi HTTP (Axios)
  4. Validation backend (Zod)
  5. Insertion MySQL
  6. Réponse JSON

3. Séparation des responsabilités

  • React → Interface utilisateur
  • Express → API serveur
  • Zod → Validation
  • MySQL → Stockage

4. Pourquoi doubler la validation

Frontend pour l’expérience utilisateur. Backend pour la sécurité.

5. Architecture logique simplifiée

Client → API → Base de données

6. Objectif pédagogique

Comprendre chaque couche indépendamment puis ensemble.

Express - Création du serveur

1. Rôle technique

Créer une API REST capable de recevoir et renvoyer du JSON.

2. Activation du mode ESM


{
  "type": "module"
}

Permet d'utiliser import/export.

3. Serveur minimal


import express from "express";

const app = express();

app.use(express.json());

app.get("/", (req, res) => {
  res.send("API active");
});

app.listen(3000);

4. Explication simplifiée

  • import → charge Express
  • express() → crée serveur
  • use() → active lecture JSON
  • get() → crée route
  • listen() → démarre serveur

5. Route POST structurée


app.post("/users", (req, res) => {
  console.log(req.body);
  res.json({ message: "Reçu" });
});

6. Vision architecture

Express est la couche contrôleur entre client et base.

Zod - Validation des données

1. Pourquoi valider

Les données utilisateur sont imprévisibles.

2. Définition d’un schéma


import { z } from "zod";

const schema = z.object({
  name: z.string().min(2),
  email: z.string().email()
});

3. Comprendre les règles

  • string → texte
  • min(2) → longueur minimale
  • email() → format valide

4. Validation sécurisée


const result = schema.safeParse(req.body);

if(!result.success){
  return res.status(400).json(result.error);
}

5. Différence parse / safeParse

  • parse → déclenche erreur
  • safeParse → retourne objet résultat

6. Intérêt stratégique

Évite d’envoyer des données corrompues en base.

MySQL - Gestion des données

1. Rôle

Stocker les données validées.

2. Connexion via pool


import mysql from "mysql2/promise";

const pool = mysql.createPool({
  host:"localhost",
  user:"root",
  password:"password",
  database:"test"
});

3. Pourquoi un pool

Optimisation des connexions.

4. Requête sécurisée


await pool.query(
 "INSERT INTO users (name,email) VALUES (?,?)",
 [name,email]
);

5. Pourquoi les ?

Protection contre injection SQL.

6. Cycle BDD

  1. Connexion
  2. Exécution
  3. Retour résultat

React Hook Form + Zod

1. Problème classique

Gestion complexe des états dans les formulaires.

2. Solution performante

React Hook Form limite les re-renders.

3. Schéma Zod côté frontend


const schema = z.object({
  name: z.string().min(2),
  email: z.string().email()
});

4. Formulaire complet


const { register, handleSubmit, formState:{errors} } =
useForm({ resolver: zodResolver(schema) });

const onSubmit = data => console.log(data);

5. Comprendre chaque élément

  • useForm → moteur
  • register → connecte input
  • handleSubmit → contrôle validation
  • errors → affiche erreur

6. Objectif UX

Valider avant envoi serveur.

Exercice pratique complet

Objectif

Créer un module d’ajout de produit.

Données requises

  • name (min 3)
  • price (positif)
  • description (min 10)

Travail backend

  • Créer schéma Zod
  • Créer route POST /products
  • Insérer en base

Travail frontend

  • Créer formulaire React Hook Form
  • Connecter Zod
  • Envoyer via Axios

Tests obligatoires

  1. Nom trop court
  2. Prix négatif
  3. Description vide
  4. Données manquantes

Extension avancée

  • Afficher liste produits
  • Ajouter suppression
  • Centraliser gestion erreurs
